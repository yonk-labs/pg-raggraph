# The A/B Test That Said "Graph Loses" — and the Empty Table That Broke It

*Or: how we almost shipped "vector always wins" off a graph with zero edges, and what the corrected test actually shows.*

> **TL;DR.** We ran an A/B test — graph retrieval vs plain vector — across three corpora and concluded **vector is never beaten, even on multi-hop data.** That conclusion was wrong. The "multi-hop" leg of the test was querying a graph that had **2,288 entities and zero relationships** — the materializer had loaded the nodes but never the edges. The recursive traversal had nothing to walk. When we re-ran on a *real* knowledge graph (9,809 typed edges) with the *real* traversal engine, graph beat vector by **~4-5pp** on multi-hop QA. The lesson isn't "graph wins" or "graph loses." It's: **check that your graph has edges before you let it testify.**

---

## The setup

[Blog 00](00-what-we-learned-building-graphrag.md) made a case that's held up well: on clean technical-doc corpora, graph mode doesn't beat plain vector + BM25. PostgreSQL docs, SEC filings, SCOTUS opinions — single-document lookup answers most questions, and a single vector ranking matches or beats the graph.

So when we built an automated A/B gate to compare a chunking pipeline's graph emission against naive vector retrieval, the early results felt familiar. On SCOTUS and NTSB — clean prose — naive won or tied. Fine. Expected.

Then a fair challenge landed: *those are exactly the corpora where you already said graph won't help. What about data built for graph — multi-hop, cross-document?*

Right. So we added [MultiHop-RAG](https://github.com/yixuantt/MultiHop-RAG) — a corpus where answers require chaining evidence across documents — ran all three retrieval modes, and reported the verdict:

> Vector is never beaten. Across all three corpora and all three modes, graph ties at best and usually loses — even on multi-hop data.

That was a clean, confident, **wrong** answer.

## The bug: a graph with no edges

Here's the query that should have been the first thing we ran, not the last:

```sql
SELECT
  (SELECT count(*) FROM entities      WHERE namespace='bakeoff-mhr-ab') AS entities,
  (SELECT count(*) FROM relationships WHERE namespace='bakeoff-mhr-ab') AS relationships;

 entities | relationships
----------+---------------
     2288 |             0
```

2,288 entities. **Zero relationships.**

pg-raggraph's graph traversal is a recursive CTE over the `relationships` table — it walks edges from one entity to the next, up to `max_hops`. With zero edges, there is nothing to walk. The "graph" modes silently degraded to a one-hop metadata lookup and we dutifully recorded the result as if it were graph retrieval. We weren't measuring "graph vs vector" on multi-hop data. We were measuring **an empty table vs vector.** It lost. Of course it lost.

The root cause was dull, which is how these things usually go. The materializer that built the A/B graph inserted entity *nodes* from the corpus, and never inserted the *edges* between them. Nodes without edges isn't a small graph — it's not a graph at all.

> **Why a "graph FAVORABLE corpus" still produced an empty graph.** The chunking pipeline emits "facts" — subject/predicate/object triples — via a dependency-parse extractor. We assumed those were a knowledge graph. They aren't. They're grammatical fragments: subjects like `He`, `They`, `It`; truncated objects like `holiday(s`; predicates that are just verbs (`drag`, `begin`, `include`). Across 2,140 facts there were 1,103 distinct subjects and 1,169 distinct objects — almost every node a singleton, and `He` in one document does not co-refer with `He` in another. Even if we *had* built edges from them, there'd be no traversable structure. That's a finding about *that emission*, not about GraphRAG.

## The corrected test, on a real graph

To actually test the thesis you need three things the broken run didn't have: a graph with **real edges**, **entity-resolved** nodes that recur across documents, and the **real recursive-traversal** retrieval path. We had all three sitting in another benchmark: `bench_musique`.

[MuSiQue](https://github.com/StonyBrookNLP/musique) is 100 compositional 2-, 3-, and 4-hop questions over ~1,700 pooled paragraphs. We'd previously ingested it with pg-raggraph's **LLM extraction**, which produces a genuine knowledge graph:

```
9,809 typed edges  ·  8,723 nodes  ·  avg degree 2.25
PART_OF · LOCATED_IN · MEMBER_OF · PARTICIPATED_IN · WON · BORDERS · …
Jim Clark --PARTICIPATED_IN--> 1967 Italian Grand Prix --WON_BY--> Honda F1 team
```

Named entities. Typed relationships. Paths that actually chain across documents. This is what the recursive traversal was built for.

The test: pure vector (`naive`) against the real recursive-traversal mode (`local`, `max_hops=2`), plus the other graph modes, on all 100 questions. Answers generated by a local Qwen model. And because a single judge can fool you, we graded every answer with **two** judges — a local gemma and gpt-5-mini — so we could measure how much of any result was real and how much was the judge.

### Two metrics, because EM is dead here

Exact-match scoring is useless on this corpus — a prior run found **27% of EM=0 answers were actually judged fully correct** (the model says "Ko Phi Phi Leh, Thailand" against a gold of "Ko Phi Phi Leh" and EM calls it wrong). So we report **support recall** (did retrieval surface the gold supporting paragraphs? — answer-model-independent) and the **LLM judge** (end-to-end answer quality, 0-3 → %).

### The result

| Mode | support recall | judge (gemma) | judge (gpt-5-mini) |
|---|:-:|:-:|:-:|
| **naive** (pure vector) | 59.3% | 40.3% | 39.0% |
| naive_boost (1-hop re-rank) | 59.3% | 42.0% | 40.7% |
| **`local`** (recursive traversal) | 57.8% | **45.3%** | **44.0%** |
| global | 58.4% | 44.0% | 38.0% |
| hybrid (vector + traversal) | 59.3% | 47.7% | 39.7% |
| smart | 56.5% | 42.7% | 41.7% |

**`local` beats `naive` by ~4-5pp on both judges** — gemma +5.0, gpt-5-mini +5.0 — and it held in a second independent answer-generation run (+3.7). On a real graph, with real traversal, **graph beats vector on multi-hop QA.** The thing Blog 00 said graph *didn't* do on technical docs, it *does* do here — because here the workload is genuinely multi-hop and the graph is genuinely a graph.

## The honest caveats (this is not a graph-wins coronation)

If Blog 00 taught us anything, it's that the second-order story matters more than the headline. Four things keep this finding honest:

**1. Retrieval recall is flat.** Every mode lands at ~58-59% support recall. Graph traversal is *not* finding more gold paragraphs than vector. The judge lift comes from surfacing them with better entity-neighborhood *context* that helps the answer model reason — a softer, smaller effect than a recall win would be.

**2. Per-hop splits are noisy, and the hop story is not what you'd guess.** With n≈33 per hop class, the breakdown wobbles:

| Comparison | 2-hop | 3-hop | 4-hop |
|---|:-:|:-:|:-:|
| `local` − `naive`, gemma | −5.1 | **+7.1** | +12.7 |
| `local` − `naive`, gpt-5-mini | +8.1 | **+5.1** | +2.0 |

**3-hop is the one consistent, trustworthy win** — both judges agree (~+5-7pp), and it held in the earlier run too. **2-hop is inconsistent** (the judges disagree on sign). **4-hop favors graph in this run on both judges, but don't trust the magnitude** — gemma says +12.7, gpt-5-mini says +2.0, and across runs gemma swung from −1.9 to +12.7 on the same 34 questions. At 4-hop, absolute accuracy is ~29% for naive; there's barely any signal to split.

**3. Only `local` is judge-robust.** gemma rates `hybrid` best (47.7%); gpt-5-mini rates it near-worst (39.7%). `global` shows the same split. The two judges agreed on only 76% of exact scores (82% binarized) — judge choice moves the ranking of every mode *except* `local`, which both judges put above naive. If you take one mode away from this, take `local`.

**4. The magnitude is modest.** +4-5pp at ~40-45% absolute accuracy. Real, repeatable, not a blowout. Anyone selling you 2× from "adding a knowledge graph" is selling.

## What actually moves the needle

The discriminating variable across this whole saga was never the retrieval algorithm. It was **graph quality**:

- Dependency-parse "facts" (`He → drag → holiday(s`) → not a traversable graph → no win, and in the broken run, no graph at all.
- LLM-extracted, entity-resolved, typed-edge KG → traversable → `local` beats vector.

So the highest-leverage investment isn't another retrieval mode. It's a **better graph**: entity resolution so nodes co-refer across documents, and typed relations so the edges mean something. The traversal engine already works — give it a real graph to walk and it earns its keep on the workload it's for.

## The meta-lesson

The embarrassing part isn't that a test was wrong. Tests are wrong all the time. The embarrassing part is that the wrong test was *confident* — three corpora, a clean verdict, a tidy table — and the thing that would have caught it was one `SELECT count(*) FROM relationships`. An empty edge table reads, in a results table, exactly like "graph doesn't help." Same number. Completely different meaning.

Before you let a graph testify that it loses, make sure it has edges.

## Run it yourself

Everything here is reproducible. Full procedure, dataset download, and the exact dual-judge command are in [`benchmarks/musique/README.md`](../../benchmarks/musique/README.md). The short version:

```bash
# 1. Build the graph (download MuSiQue, sample 100 Q, pool corpus, ingest with LLM extraction)
python3 benchmarks/musique/prepare.py
uv run python benchmarks/musique/ingest.py

# 2. The graph-vs-vector run: 6 modes, dual judge (gemma + gpt-5-mini)
export PGRG_DSN="postgresql://postgres:postgres@localhost:5434/pg_raggraph"
export PGRG_TEST_LLM_URL="http://<answer-model>/v1"          # answer generation
export PGRG_JUDGE_LLM_URL="http://<judge-model>/v1"          # judge, decoupled from answer model
export PGRG_JUDGE_LLM_MODEL="<judge-model-id>"
export OPENAI_API_KEY="…"; export OPENAI_JUDGE_MODEL="gpt-5-mini"   # second judge

uv run python benchmarks/musique/run.py \
  --modes naive,naive_boost,local,global,hybrid,smart \
  --judge both --out-tag full-6mode-dual-judge

# 3. Sanity check FIRST — confirm the graph actually has edges:
psql "$PGRG_DSN" -c \
  "SELECT count(*) FROM relationships WHERE namespace='bench_musique';"   # expect ~9,809, not 0
```

Raw results: `benchmarks/musique/_results/results-full-6mode-dual-judge.json`. Corrected A/B writeup with the full retraction: [`benchmarks/ab-gate/RESULTS.md`](../../benchmarks/ab-gate/RESULTS.md).

*Benchmarks run 2026-05-29. pg-raggraph `0.5.0a5`. Judges: gemma-4-26B (local) + gpt-5-mini. Answer model: Qwen3-Coder-Next (local). n=100 MuSiQue questions, stratified 33/33/34 across 2/3/4-hop.*
