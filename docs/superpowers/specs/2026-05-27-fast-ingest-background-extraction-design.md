# Fast Ingest + Background Extraction — Design & Handoff (2026-05-27)

## TL;DR

Today's gate proved that on MHR, graph traversal gives **zero retrieval lift** over `naive` even with 84.5k LLM-extracted relationships — and costs 1.4–2.2× the latency. The fast-path ingest probe pinpointed embedding as the **only meaningful cost** in `skip_llm` ingest (99%, ~140 ms/chunk on bge-small CPU). Conclusion: **the LLM extraction step is off the critical path** for queryability and should be decoupled from `ingest()`. This doc designs and plans (a) a Postgres-backed background extraction queue with both off-server CLI and in-process daemon surfaces, then (b) embedding-alternative measurements once the fast-path floor is the only remaining bottleneck.

## Mission Brief

### Purpose
Decouple LLM relationship extraction from the synchronous `ingest()` call, so a document becomes `naive`-queryable immediately and the graph backfills asynchronously. Provide two operator surfaces (off-server CLI + in-process daemon) so the same primitive serves CRON-driven backfills and long-running services. Make ingestion a "non-event."

### Desired Outcome
A user calling `GraphRAG.ingest()` returns in **chunking + embedding time only** (~0.6 s/doc on local CPU embedder; faster with shared/GPU embeddings). The document is queryable via `naive` immediately. A background process — invokable either as `pgrg extract` (off-server) or as a daemon thread inside a long-running app — drains a per-document extraction queue with explicit loop control (batch size, rate limit, max iterations, graceful shutdown). Querying `local`/`global`/`hybrid` modes against a not-yet-extracted document degrades gracefully (returns what's available; surfaces a `graph_status` hint in `QueryResult.metadata` so callers can choose to wait or retry).

### Success Criteria
- **SC-001** — A new `documents.graph_status` column tracks lifecycle: `'pending' | 'processing' | 'ready' | 'failed'`. Indexed for queue polling. Migration is reversible (or, if forward-only, documented).
- **SC-002** — Default `ingest()` behavior changes from "extract synchronously" to "store as `pending`." Existing tests pass with a one-line opt-in to the legacy synchronous behavior (e.g., `ingest(..., extract='sync')` or `defer_extraction=False`), so consumers aren't broken silently.
- **SC-003** — A `Database.next_pending_documents(batch_size, namespace=None)` (or equivalent) uses `SELECT ... FOR UPDATE SKIP LOCKED` to atomically claim N documents; concurrent workers never claim the same doc. Verified with a multi-worker race test.
- **SC-004** — A pure `extract_documents(doc_ids)` primitive runs the existing extraction pipeline against the claimed batch, writes entities/relationships in one transaction per doc, and updates `graph_status` to `ready` (or `failed` with an error reason). Idempotent: re-running on a doc already `ready` is a no-op; re-running on a `failed` doc retries.
- **SC-005** — A new CLI subcommand `pgrg extract` exposes loop control flags: `--batch-size N` (default 4), `--max-iterations M` (default unlimited), `--rate-limit-rps R` (default unlimited), `--namespace NS` (default all), `--once` (single iteration), and `--daemon` (long-running with SIGTERM-graceful shutdown). Exits 0 when the queue drains (without `--daemon`).
- **SC-006** — Query path: `local`/`global`/`hybrid` return successfully against a namespace with `pending` docs (just with fewer entities/relationships); `QueryResult.metadata` exposes `graph_status_summary` (counts by status) so callers can choose to wait. No exceptions, no silent empty results.
- **SC-007** — Tests: a unit test for the SKIP-LOCKED claim function (multi-process race); an integration test that ingests pending → extracts via CLI → re-queries to see the graph populate; a regression test that legacy synchronous ingest still works behind the opt-in.
- **SC-008** — Documentation: `docs/cookbook/background-extraction.md` walks both surfaces, and `CLAUDE.md` is updated to mention the new default and the two surfaces.
- **SC-009** — Embedding-bottleneck probes (Section "Then: Embedding Measurements") produce a comparison table: local CPU bge-small vs at least one alternative (shared HTTP / GPU / different model), with cold vs warm-cache numbers for each.

## Architecture

### Status model
Add `documents.graph_status TEXT` with a CHECK constraint or app-level discipline (4 values). Index `(namespace, graph_status)` for queue polling. Two related columns help debuggability and shouldn't add scope: `graph_extracted_at TIMESTAMPTZ` (set when transitioning to `ready`) and `graph_error TEXT NULL` (the failure reason on `failed`).

### Queue mechanism (Postgres SKIP LOCKED — no broker)
```sql
SELECT id, content_hash
FROM documents
WHERE namespace = COALESCE($1, namespace)
  AND graph_status = 'pending'
ORDER BY created_at
LIMIT $2
FOR UPDATE SKIP LOCKED;
```
Inside the same transaction: `UPDATE documents SET graph_status = 'processing' WHERE id = ANY($claimed)`, commit. Workers can run concurrently safely; `SKIP LOCKED` makes claimed rows invisible to peers. This is the canonical Postgres queue pattern and fits the single-DB thesis exactly.

`★ Pattern note ─────────────────────────────────`
`SKIP LOCKED` was added in Postgres 9.5 for exactly this use case. Unlike advisory locks (which we got bitten by in #45), `FOR UPDATE SKIP LOCKED` is row-scoped, transaction-scoped, and doesn't interact with CONCURRENTLY index builds. Workers compete cleanly without any lock orchestration in application code.
`─────────────────────────────────────────────────`

### Two surfaces on one primitive
The core is `extract_documents(doc_ids: list[int]) -> ExtractStats`. Build it pure (no CLI, no signal handling, no loop) so it's unit-testable and reusable.

- **CLI** (`pgrg extract`): a thin loop around `claim → extract → repeat`, with flags translating to loop config. Exits when queue drains (or `--max-iterations` reached, or `--once`).
- **Daemon** (`pgrg extract --daemon`): same loop, but runs forever with SIGTERM/SIGINT handlers that finish the current batch then exit cleanly. Implement via `asyncio.Event` for cooperative shutdown.

### Idempotency
Per-document, single-transaction. The transaction does: extract → resolve entities → insert entities & relationships → update `graph_status='ready'` + `graph_extracted_at=now()`. On exception: rollback → update `graph_status='failed'` + `graph_error=str(e)` in a separate small transaction. Re-running `extract` on a `failed` doc: the claim filter includes `pending` only; add `--include-failed` flag to retry failed ones.

## Then: Embedding-Bottleneck Measurements

Background-extraction takes the LLM out of the critical path; once it's out, the fast-path floor is embedding (140 ms/chunk on local CPU bge-small). Measure alternatives so the operator knows the levers.

### What to measure
For each variant, time **cold ingest of 40 MHR docs / ~187 chunks** (same harness as `benchmarks/ingest_perf.py`):
1. **Current baseline** — local fastembed bge-small-en-v1.5 (CPU). Already measured: 140 ms/chunk.
2. **Shared HTTP embedding endpoint** — spin up a TEI (Text Embeddings Inference) container locally, or use an OpenAI-compatible embedding endpoint. Measure with HTTP overhead included.
3. **Larger model (bge-large)** — same provider, larger model. Trade quality for speed; we want to see the speed delta to inform the "embedding pool" tier decision.
4. **GPU embedding** (if available on this box) — same model, GPU device. Establishes the floor.
5. **Batch size sweep** — keep current model, vary `fastembed` batch size (16, 32, 64, 128). The current default may be sub-optimal for this hardware.

### How to set up the shared HTTP endpoint
```bash
docker run -p 8081:80 --pull always \
  ghcr.io/huggingface/text-embeddings-inference:cpu-latest \
  --model-id BAAI/bge-small-en-v1.5
# Then point pg-raggraph at it:
export PGRG_EMBEDDING_PROVIDER=http
export PGRG_EMBEDDING_BASE_URL=http://localhost:8081
```
Verify with a one-shot embed call before the timing run.

### What to report
A table per variant, same columns as the existing probe (chunking, embedding cold, store, total, throughput). Plus a recommendation: which variant is the right default for which user (laptop/dev vs. dedicated server vs. shared-tenant service).

### Anti-pattern to avoid
Don't measure with the embedding cache warm — you'll learn what re-ingest costs (already known: 1.8 ms/chunk), not what the embedder costs. The probe already clears the cache between cold/warm runs; preserve that.
